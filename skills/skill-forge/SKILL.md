# Skill Forge — 스킬 생성/테스트/개선 메타-스킬

## 1. 핵심 원리

Anthropic의 공식 Skill Creator 패턴을 기반으로:
- **Progressive Disclosure**: 메타데이터(항상 로드) → 본문(트리거 시) → 리소스(필요 시)
- **Eval-Driven Development**: 만들고 → 테스트하고 → 비교하고 → 개선하는 루프
- **Description이 트리거**: description 필드가 스킬 발동의 핵심. "pushy"하게 작성.

## 2. 스킬 생성 워크플로우

```
[의도 캡처] → [인터뷰] → [SKILL.md 작성] → [테스트] → [평가] → [개선] → [패키징]
```

### 2.1 의도 캡처

사용자에게 4가지 질문:

1. **이 스킬이 Claude에게 뭘 하게 할 건가요?**
2. **언제 발동해야 하나요?** (사용자가 할 말/상황)
3. **출력 형식은?** (파일, 터미널 출력, 대화 등)
4. **테스트 케이스를 만들까요?** (권장: Yes)

### 2.2 인터뷰 (심화)

의도를 기반으로 추가 질문:
- 엣지 케이스는?
- 입출력 형식의 구체적 예시?
- 성공 기준은?
- 사용할 외부 도구/MCP는?
- 기존 스킬과 겹치는 부분은?

### 2.3 SKILL.md 작성

두 가지 형식 중 선택:

**A. Skill (SKILL.md)** — 자동 트리거, 복잡한 워크플로우:
```
~/.claude/skills/스킬명/SKILL.md
```

**B. Command (commands/*.md)** — 수동 /슬래시 명령:
```
~/.claude/commands/명령명.md
```

#### SKILL.md 작성 규칙:

```markdown
---
name: 스킬-이름
description: "설명. 트리거 키워드를 충분히 나열. 약간 pushy하게."
---

# 스킬 제목

## 1. 핵심 철학
왜 이 스킬이 존재하는가.

## 2. 워크플로우
단계별 실행 흐름.

## 3. 상세 규칙
각 단계의 구체적 행동.

## 4. 출력 템플릿
정확한 출력 형식.

## 5. 에러 핸들링
예상 가능한 문제와 대응.
```

**작성 원칙:**
- 명령형으로 작성 ("~해라", "~하지 말라")
- WHY를 설명 (MUST/NEVER 남발 대신 이유 제시)
- 500줄 이하 유지. 길면 `references/` 로 분리.
- 구체적 예시(Input/Output 쌍) 포함.
- 한 스킬이 너무 많은 걸 하지 않게.

**Description 최적화:**
- 스킬이 하는 일 + 언제 발동할지를 모두 포함.
- 사용자가 실제로 할 말을 키워드로 나열.
- Claude는 혼자 쉽게 할 수 있는 작업에는 스킬을 안 쓰므로, 스킬이 필요한 복잡한 시나리오를 강조.

### 2.4 테스트 케이스 작성

2-3개의 현실적 테스트 프롬프트 작성:

```json
// evals/evals.json
[
  {
    "id": "eval-01",
    "prompt": "실제 사용자가 할 법한 말",
    "assertions": [
      "출력에 X가 포함되어야 함",
      "파일 Y가 생성되어야 함",
      "Z 형식이어야 함"
    ]
  }
]
```

**좋은 테스트의 조건:**
- 실제 사용자가 할 법한 자연스러운 말
- 다양한 시나리오 커버 (정상, 엣지, 에러)
- 검증 가능한 구체적 기대 결과

### 2.5 병렬 평가

각 테스트 케이스에 대해 Agent 도구로 2개 병렬 실행:

1. **with-skill**: 새 스킬 적용 상태에서 실행
2. **without-skill (baseline)**: 스킬 없이 실행

결과를 비교해서 스킬이 실제로 품질을 올렸는지 확인.

### 2.6 평가 및 개선

비교 결과를 사용자에게 제시:
- 각 테스트별 with/without 결과 나란히 보여주기
- assertions 통과율
- 스킬이 확실히 나아진 점 / 오히려 나빠진 점

사용자 피드백 → SKILL.md 수정 → 재테스트 (iteration-N+1).

### 2.7 패키징

완성된 스킬의 최종 점검:
- [ ] description이 충분히 pushy한가
- [ ] 워크플로우가 명확한가
- [ ] 에러 핸들링이 있는가
- [ ] 테스트를 통과했는가
- [ ] 기존 스킬과 충돌하지 않는가

## 3. 번들 리소스 구조

복잡한 스킬은 추가 리소스를 포함:

```
skill-name/
├── SKILL.md              # 핵심 지시문 (<500줄)
├── scripts/              # 실행 가능한 코드 (결정적 작업)
├── references/           # 참고 문서 (필요 시 로드)
├── assets/               # 템플릿, 폰트, 아이콘
└── evals/
    └── evals.json        # 테스트 케이스
```

- `scripts/`: Claude가 실행하되 컨텍스트에 로드하지 않아도 되는 코드.
- `references/`: SKILL.md에서 "references/X.md를 참조하라"고 포인터만 두기.
- `assets/`: 출력에 사용되는 정적 파일.

## 4. 기존 스킬 개선

이미 있는 스킬을 개선할 때:

1. 현재 SKILL.md 읽기.
2. 문제점 또는 개선 요청 파악.
3. 테스트 케이스 작성 (before/after 비교용).
4. 수정 적용.
5. 병렬 평가로 실제 개선 확인.
6. 반복.

## 5. 스킬 목록 관리

현재 설치된 스킬 확인:
```bash
ls ~/.claude/skills/      # 자동 트리거 스킬
ls ~/.claude/commands/     # /슬래시 커맨드
```

스킬 간 충돌 체크: description 키워드가 겹치면 우선순위 문제 발생 가능.
