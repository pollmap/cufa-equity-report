# Get Shit Done (GSD) — 컨텍스트 엔지니어링 워크플로우

## 1. 핵심 원리

**Context Rot 문제**: Claude가 긴 세션에서 컨텍스트 윈도우가 차면 품질이 급락함.
**해결책**: 작업을 원자적 플랜으로 분해 → 각 플랜을 독립 서브에이전트(신선한 컨텍스트)로 실행.

## 2. 모드 선택

| 모드 | 언제 | 명령 |
|------|------|------|
| **Full** | 대규모 기능, 리팩토링, 신규 프로젝트 | `/gsd` 또는 "GSD로 해줘" |
| **Quick** | 작은 버그, 설정 변경, 단순 추가 | `/gsd:quick` 또는 "빠르게 해줘" |

## 3. Full 워크플로우 (5단계)

```
[discuss] → [plan] → [execute] → [verify] → [ship]
```

### 3.1 Discuss — 요구사항 구체화

1. 사용자의 요청을 듣고 핵심 질문 3-5개로 구체화.
2. 구현 방향에 대한 합의 도출.
3. 비기능 요구사항 확인 (성능, 보안, 호환성).
4. 결과를 `.planning/REQUIREMENTS.md`에 저장.

```markdown
# REQUIREMENTS.md
## 목표
## 제약 조건
## 비기능 요구사항
## 합의된 구현 방향
```

### 3.2 Plan — 리서치 + 원자적 플랜 생성

1. **리서치 에이전트 4개 병렬 실행**: 코드베이스 탐색, 의존성 분석, 패턴 파악, 엣지케이스 조사.
2. 리서치 결과를 `.planning/RESEARCH.md`에 통합.
3. 작업을 **원자적 플랜**으로 분해:
   - 각 플랜은 독립적으로 실행 가능
   - 각 플랜은 하나의 커밋 단위
   - 의존성 그래프 명시

4. `.planning/PLAN.md` 생성:

```markdown
# PLAN.md

## Wave 1 (병렬 실행 가능)
### Plan 01: [제목]
- 파일: path/to/file.py
- 변경: 무엇을 어떻게
- 테스트: 검증 방법
- 커밋: feat(01): 설명

### Plan 02: [제목]
...

## Wave 2 (Wave 1 완료 후)
### Plan 03: [제목]
- 의존: Plan 01, 02
...
```

### 3.3 Execute — 웨이브 기반 병렬 실행

1. 같은 Wave의 플랜들을 **Agent 도구로 병렬 실행**.
2. 각 에이전트는 해당 플랜만 받아서 독립 실행 (신선한 컨텍스트).
3. 에이전트에게 전달하는 정보:
   - 해당 플랜의 상세 내용
   - 관련 파일 경로
   - 코딩 컨벤션
   - 테스트 방법
4. Wave 완료 후 결과 통합, 다음 Wave 진행.
5. 각 플랜 완료 시 원자적 커밋: `feat(01-01): 설명`

### 3.4 Verify — 검증

1. 전체 변경사항 리뷰 (git diff).
2. 테스트 실행.
3. 사용자 수동 검증 항목 제시.
4. 문제 발견 시 해당 플랜만 재실행.

### 3.5 Ship — 배포

1. 커밋 히스토리 정리.
2. PR 생성 또는 사용자에게 배포 방법 안내.
3. `.planning/SUMMARY.md`에 완료 보고서 저장.

## 4. Quick 모드

큰 세레모니 없이 바로 실행:

1. 요청 파악 → 관련 파일 읽기
2. 변경 적용
3. 검증
4. 커밋 (선택)

`--research` 플래그: 방향이 불확실할 때 리서치 에이전트 1개 선행.
`--discuss` 플래그: 그레이 영역이 있을 때 질문 먼저.

## 5. .planning/ 디렉토리 관리

```
.planning/
├── PROJECT.md        # 프로젝트 개요 (코드베이스 맵)
├── REQUIREMENTS.md   # discuss 단계 산출물
├── RESEARCH.md       # 리서치 통합 결과
├── PLAN.md           # 원자적 플랜 + 의존성 그래프
├── STATE.md          # 현재 진행 상태
└── SUMMARY.md        # 완료 보고서
```

- `.gitignore`에 `.planning/`을 추가할지 사용자에게 확인.
- 다음 GSD 사이클 시작 시 이전 `.planning/`은 아카이브.

## 6. 컨텍스트 관리 원칙

- 메인 컨텍스트는 **30-40%** 이하 유지.
- 무거운 작업은 전부 서브에이전트에게 위임.
- 서브에이전트에게는 필요한 정보만 정확히 전달 (전체 히스토리 X).
- 파일 내용은 직접 전달하지 말고 경로만 전달 → 에이전트가 직접 읽게 함.

## 7. 원자적 커밋 규칙

```
feat(WW-PP): 설명    # WW=웨이브, PP=플랜 번호
fix(WW-PP): 설명
refactor(WW-PP): 설명
test(WW-PP): 설명
```

각 커밋은 독립적으로 revert 가능해야 함.
